## gRPC Streaming Patterns

### Introduction

Standard gRPC uses **unary** RPCs — one request in, one response out. But gRPC also supports **streaming**, where the client, the server, or both can send a continuous flow of messages over a single RPC connection.

This lesson covers all four patterns, how they differ, and when to use each.

---

## The Four gRPC Communication Patterns

### 1. Unary RPC (Request-Response)

The simplest pattern — identical to a regular function call.

```
Client                          Server
  │                               │
  │──── Request ─────────────────►│
  │                               │  (process)
  │◄──── Response ────────────────│
  │                               │
```

**Proto syntax:**
```protobuf
rpc PlaceOrder(OrderRequest) returns (OrderResponse);
```

**When to use:** Simple CRUD operations, lookups, single actions.

---

### 2. Server Streaming RPC

Client sends **one** request, server sends back **a stream** of responses.

```
Client                          Server
  │                               │
  │──── Request ─────────────────►│
  │                               │
  │◄──── Response 1 ──────────────│
  │◄──── Response 2 ──────────────│
  │◄──── Response 3 ──────────────│
  │◄──── ...        ──────────────│
  │◄──── Response N ──────────────│
  │◄──── (completed) ─────────────│
  │                               │
```

**Proto syntax — note `stream` on the return type:**
```protobuf
rpc StreamPrices(PriceRequest) returns (stream PriceTick);
```

**When to use:**
- Live price feeds / stock tickers
- Real-time logs and event streams
- Download large datasets in chunks
- Push notifications

**Java server side:**
```java
@Override
public void streamPrices(PriceRequest request,
                         StreamObserver<PriceTick> responseObserver) {
    // Send multiple responses
    for (int i = 0; i < 100; i++) {
        responseObserver.onNext(buildPriceTick(i));
    }
    responseObserver.onCompleted(); // signal end of stream
}
```

**Java client side:**
```java
// Blocking iterator — reads responses one by one
Iterator<PriceTick> prices = blockingStub.streamPrices(request);
while (prices.hasNext()) {
    PriceTick tick = prices.next();
    System.out.println("Price: " + tick.getPrice());
}
```

---

### 3. Client Streaming RPC

Client sends **a stream** of messages, server responds with **one** response after processing all of them.

```
Client                          Server
  │                               │
  │──── Message 1 ───────────────►│
  │──── Message 2 ───────────────►│
  │──── Message 3 ───────────────►│
  │──── ...        ───────────────►│
  │──── Message N ───────────────►│
  │──── (completed) ──────────────►│
  │                               │  (process all)
  │◄──── Response ────────────────│
  │                               │
```

**Proto syntax — note `stream` on the request type:**
```protobuf
rpc BatchPlaceOrders(stream OrderRequest) returns (BatchOrderResponse);
```

**When to use:**
- Batch uploads (files, records)
- Aggregation (client sends data points, server returns summary)
- Bulk order submission

**Java server side:**
```java
@Override
public StreamObserver<OrderRequest> batchPlaceOrders(
        StreamObserver<BatchOrderResponse> responseObserver) {

    return new StreamObserver<OrderRequest>() {
        List<OrderRequest> orders = new ArrayList<>();

        @Override
        public void onNext(OrderRequest order) {
            orders.add(order); // accumulate incoming messages
        }

        @Override
        public void onCompleted() {
            // Process all collected orders
            BatchOrderResponse response = processAll(orders);
            responseObserver.onNext(response);
            responseObserver.onCompleted();
        }

        @Override
        public void onError(Throwable t) {
            System.err.println("Client error: " + t.getMessage());
        }
    };
}
```

**Java client side:**
```java
// Async stub required for client streaming
StreamObserver<OrderRequest> requestStream =
    asyncStub.batchPlaceOrders(responseObserver);

// Send multiple messages
requestStream.onNext(order1);
requestStream.onNext(order2);
requestStream.onNext(order3);
requestStream.onCompleted(); // signal end of stream
```

### 4. Bidirectional Streaming RPC ⭐

Both client and server send **independent streams** of messages. The two streams operate **independently** — the client and server can read and write in any order.

```
Client                          Server
  │                               │
  │──── Message A1 ──────────────►│
  │◄──── Message B1 ──────────────│
  │──── Message A2 ──────────────►│
  │──── Message A3 ──────────────►│
  │◄──── Message B2 ──────────────│
  │◄──── Message B3 ──────────────│
  │──── Message A4 ──────────────►│
  │◄──── Message B4 ──────────────│
  │──── (completed) ──────────────►│
  │◄──── Message B5 ──────────────│
  │◄──── (completed) ─────────────│
  │                               │
```

**Proto syntax — `stream` on BOTH request and return:**
```protobuf
rpc LiveTrading(stream TradeCommand) returns (stream ExecutionReport);
```

**Critical characteristics:**
- Client and server streams are **fully independent**
- Either side can send messages at **any time**
- Either side can close its stream independently
- Server **does not** wait for all client messages before responding
- Messages are **not** paired — server may send 3 responses for 1 client message, or vice versa

**When to use:**
- Real-time chat
- Live trading (send orders, receive executions simultaneously)
- Multiplayer game state sync
- Collaborative editing
- IoT sensor data with real-time alerts

## Streaming Pattern Comparison

| Aspect | Unary | Server Stream | Client Stream | Bidirectional |
|--------|-------|---------------|---------------|---------------|
| **Client stub type** | Blocking or Async | Blocking or Async | **Async only** | **Async only** |
| **Client sends** | 1 message | 1 message | N messages | N messages |
| **Server sends** | 1 message | N messages | 1 message | N messages |
| **Streams independent?** | N/A | N/A | N/A | ✅ Yes |
| **Ordering guaranteed?** | N/A | ✅ Yes (per stream) | ✅ Yes (per stream) | ✅ Yes (per stream) |
| **Back-pressure** | N/A | Automatic | Automatic | Automatic |

## Key Java Types for Streaming

| Type | Role |
|------|------|
| `StreamObserver<T>` | Core interface — both sending and receiving use this |
| `StreamObserver.onNext(T)` | Send a message |
| `StreamObserver.onCompleted()` | Signal end of stream |
| `StreamObserver.onError(Throwable)` | Signal an error |
| `BlockingStub` | Synchronous client — supports unary and server streaming |
| `Stub` (async) | Asynchronous client — supports **all four** patterns |
| `CountDownLatch` | Synchronization primitive to wait for async completion |

> **Important:** Client streaming and bidirectional streaming require the **async stub** (`newStub()`), not the blocking stub (`newBlockingStub()`). This is because the client needs to send messages asynchronously while potentially receiving responses.
