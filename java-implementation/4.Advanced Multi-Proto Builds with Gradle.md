## Advanced: Multi-Proto Builds with Gradle

### Introduction

In real-world gRPC projects, you rarely have a single `.proto` file. Large systems decompose their protobuf definitions across **multiple files** that **import** from one another — just like how Java classes import from shared packages.

This lesson covers:

- How proto `import` statements work
- Organizing **common/shared** proto files
- Building a **multi-service** protobuf architecture
- Gradle configuration for multi-proto projects
- A complete, large-scale example

---

## Proto Imports & Packages

### The `import` Statement

Proto files can import definitions from other `.proto` files using the `import` keyword:

```protobuf
import "common/base.proto";
import "common/user.proto";
```

**Key rules:**
* Import paths are **relative** to the proto source root (`src/main/proto/`)
* The imported file must be on the protoc compiler's search path
* Circular imports are **not allowed**

### The `package` Statement

Proto packages prevent naming collisions — similar to Java packages:

```protobuf
package common;

message Timestamp { ... }
```

When referencing a type from another package, use the fully-qualified name:

```protobuf
import "common/base.proto";

message Order {
  common.Timestamp created_at = 1;  // fully-qualified reference
}
```

### `java_package` vs `package`

| Directive | Purpose |
|-----------|---------|
| `package` | Proto-level namespace — controls how **other proto files** reference your types |
| `java_package` | Java-level namespace — controls the **generated Java package** |

They are independent. You typically set both:

```protobuf
package common;                              // proto namespace
option java_package = "com.example.grpc.common";  // Java package
```

## Project Structure for Multi-Proto Builds

A well-organized multi-proto project mirrors the package hierarchy:

```
grpc-multi-proto/
├── build.gradle
├── settings.gradle
└── src/
    └── main/
        ├── java/
        │   └── com/example/grpc/
        │       ├── server/
        │       │   └── ECommerceServer.java
        │       └── client/
        │           ├── OrderClient.java
        │           ├── InventoryClient.java
        │           └── NotificationClient.java
        └── proto/
            ├── common/
            │   ├── base.proto            ← shared primitives
            │   ├── user.proto            ← shared user types
            │   └── pagination.proto      ← shared pagination
            ├── orders/
            │   └── order_service.proto   ← imports common/*
            ├── inventory/
            │   └── inventory_service.proto  ← imports common/*
            └── notifications/
                └── notification_service.proto  ← imports common/* and orders/*
```

**Design principles:**
1. **`common/`** — Types reused by multiple services live here
2. **Service folders** — Each service has its own folder and proto file
3. **Import direction** — Services import from `common/`, never the reverse
4. **Dependency layering** — A service can import another service's types (e.g., notifications imports orders) but avoid circular dependencies

## Gradle Configuration

The standard protobuf Gradle plugin handles multi-proto builds **automatically** — all `.proto` files under `src/main/proto/` are compiled together, so imports resolve naturally.

```groovy
plugins {
    id 'java'
    id 'application'
    id 'com.google.protobuf' version '0.9.4'
}

group = 'com.example'
version = '1.0-SNAPSHOT'
sourceCompatibility = '11'

repositories {
    mavenCentral()
}

def grpcVersion = '1.68.0'
def protobufVersion = '4.28.0'

dependencies {
    implementation "io.grpc:grpc-netty-shaded:${grpcVersion}"
    implementation "io.grpc:grpc-protobuf:${grpcVersion}"
    implementation "io.grpc:grpc-stub:${grpcVersion}"
    compileOnly 'javax.annotation:javax.annotation-api:1.3.2'
}

protobuf {
    protoc {
        artifact = "com.google.protobuf:protoc:${protobufVersion}"
    }
    plugins {
        grpc {
            artifact = "io.grpc:protoc-gen-grpc-java:${grpcVersion}"
        }
    }
    generateProtoTasks {
        all()*.plugins {
            grpc {}
        }
    }
}

// Custom run tasks for each server/client
task runServer(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.server.ECommerceServer'
}

task runOrderClient(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.client.OrderClient'
}

task runInventoryClient(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.client.InventoryClient'
}

task runNotificationClient(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.client.NotificationClient'
}
```

> **Note:** No special configuration is needed for imports — the plugin compiles all `.proto` files in `src/main/proto/` as a single unit, so `import "common/base.proto"` resolves automatically.

### Proto Files

#### `common/base.proto` — Shared Primitives

This file contains fundamental types reused across the entire platform.

```protobuf
syntax = "proto3";

package common;

option java_package = "com.example.grpc.common";
option java_outer_classname = "BaseProto";
option java_multiple_files = true;

// Represents a point in time — used by every service
message Timestamp {
  int64 seconds = 1;    // Unix epoch seconds
  int32 nanos = 2;      // Sub-second precision
}

// Represents a monetary value with currency
message Money {
  string currency_code = 1;  // ISO 4217 (e.g., "USD", "EUR", "INR")
  int64 units = 2;           // Whole units (e.g., 42 for $42.99)
  int32 nanos = 3;           // Nano units (e.g., 990000000 for $42.99)
}

// Standardized error response across all services
message ErrorResponse {
  int32 code = 1;
  string message = 2;
  string details = 3;
  Timestamp timestamp = 4;   // self-referencing within the same file
}

// Common status enum reusable across services
enum OperationStatus {
  OPERATION_STATUS_UNSPECIFIED = 0;
  OPERATION_STATUS_SUCCESS = 1;
  OPERATION_STATUS_FAILURE = 2;
  OPERATION_STATUS_PENDING = 3;
}

// Generic key-value metadata
message Metadata {
  map<string, string> attributes = 1;
}
```

**Design notes:**
* `java_multiple_files = true` generates each message as a **separate Java class** instead of nesting them inside an outer class — cleaner imports for large projects
* `Timestamp` and `Money` are the most commonly shared types
* `ErrorResponse` itself uses `Timestamp` (self-reference within the same file — no import needed)
* Enum values are prefixed with the enum name (`OPERATION_STATUS_`) per proto3 best practices

#### `common/user.proto` — Shared User Types

User types are referenced by orders, notifications, and potentially other services.

```protobuf
syntax = "proto3";

package common;

import "common/base.proto";

option java_package = "com.example.grpc.common";
option java_outer_classname = "UserProto";
option java_multiple_files = true;

// Lightweight user reference — avoids duplicating full user data
message UserInfo {
  string user_id = 1;
  string username = 2;
  string email = 3;
  UserRole role = 4;
}

enum UserRole {
  USER_ROLE_UNSPECIFIED = 0;
  USER_ROLE_CUSTOMER = 1;
  USER_ROLE_ADMIN = 2;
  USER_ROLE_SUPPORT = 3;
}

// Full address — reused by orders (shipping) and users (billing)
message Address {
  string street_line_1 = 1;
  string street_line_2 = 2;
  string city = 3;
  string state = 4;
  string postal_code = 5;
  string country_code = 6;  // ISO 3166-1 alpha-2 (e.g., "US", "IN")
  Metadata labels = 7;      // ← uses common.Metadata from base.proto
}
```

**Import pattern:** `user.proto` imports `base.proto` to use `Metadata`. Since both files are in the `common` package, the reference is simply `Metadata` (no prefix needed).

#### `common/pagination.proto` — Shared Pagination

Standard pagination patterns used by any List/Search RPC.

```protobuf
syntax = "proto3";

package common;

option java_package = "com.example.grpc.common";
option java_outer_classname = "PaginationProto";
option java_multiple_files = true;

// Client sends this to request a specific page
message PageRequest {
  int32 page_number = 1;   // 1-indexed
  int32 page_size = 2;     // items per page (max 100)
  string sort_by = 3;      // field name to sort by
  bool ascending = 4;      // sort direction
}

// Server includes this in paginated responses
message PageResponse {
  int32 total_items = 1;
  int32 total_pages = 2;
  int32 current_page = 3;
  bool has_next = 4;
  bool has_previous = 5;
}
```

**No imports needed** — this file is self-contained but still in the `common` package so other files can import it.

#### `orders/order_service.proto` — Order Service

This service imports from **three** common files.

```protobuf
syntax = "proto3";

package orders;

import "common/base.proto";
import "common/user.proto";
import "common/pagination.proto";

option java_package = "com.example.grpc.orders";
option java_outer_classname = "OrderServiceProto";
option java_multiple_files = true;

// ─────────────────────────────────────────────────
// Messages
// ─────────────────────────────────────────────────

message OrderItem {
  string product_id = 1;
  string product_name = 2;
  int32 quantity = 3;
  common.Money unit_price = 4;     // ← from base.proto
  common.Money total_price = 5;    // ← from base.proto
}

message Order {
  string order_id = 1;
  common.UserInfo customer = 2;           // ← from user.proto
  common.Address shipping_address = 3;    // ← from user.proto
  OrderStatus status = 4;
  repeated OrderItem items = 5;
  common.Money subtotal = 6;             // ← from base.proto
  common.Money tax = 7;                  // ← from base.proto
  common.Money total = 8;               // ← from base.proto
  common.Timestamp created_at = 9;      // ← from base.proto
  common.Timestamp updated_at = 10;     // ← from base.proto
  common.Metadata metadata = 11;        // ← from base.proto
  PaymentMethod payment_method = 12;
}

enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
  ORDER_STATUS_CONFIRMED = 2;
  ORDER_STATUS_PROCESSING = 3;
  ORDER_STATUS_SHIPPED = 4;
  ORDER_STATUS_DELIVERED = 5;
  ORDER_STATUS_CANCELLED = 6;
  ORDER_STATUS_REFUNDED = 7;
}

enum PaymentMethod {
  PAYMENT_METHOD_UNSPECIFIED = 0;
  PAYMENT_METHOD_CREDIT_CARD = 1;
  PAYMENT_METHOD_DEBIT_CARD = 2;
  PAYMENT_METHOD_UPI = 3;
  PAYMENT_METHOD_NET_BANKING = 4;
  PAYMENT_METHOD_CASH_ON_DELIVERY = 5;
}

// ─────────────────────────────────────────────────
// Request / Response Messages
// ─────────────────────────────────────────────────

message CreateOrderRequest {
  common.UserInfo customer = 1;
  common.Address shipping_address = 2;
  repeated OrderItem items = 3;
  PaymentMethod payment_method = 4;
}

message GetOrderRequest {
  string order_id = 1;
}

message ListOrdersRequest {
  string customer_id = 1;             // filter by customer
  OrderStatus status_filter = 2;     // filter by status
  common.PageRequest pagination = 3;  // ← from pagination.proto
}

message ListOrdersResponse {
  repeated Order orders = 1;
  common.PageResponse pagination = 2;  // ← from pagination.proto
}

message UpdateOrderStatusRequest {
  string order_id = 1;
  OrderStatus new_status = 2;
}

// ─────────────────────────────────────────────────
// Service Definition
// ─────────────────────────────────────────────────

service OrderService {
  // Create a new order
  rpc CreateOrder(CreateOrderRequest) returns (Order);

  // Get a single order by ID
  rpc GetOrder(GetOrderRequest) returns (Order);

  // List orders with filtering and pagination
  rpc ListOrders(ListOrdersRequest) returns (ListOrdersResponse);

  // Update order status (e.g., confirm, ship, cancel)
  rpc UpdateOrderStatus(UpdateOrderStatusRequest) returns (Order);
}
```

**Cross-package references:** Since the order service is in `package orders` and imports types from `package common`, it must use the fully-qualified `common.Money`, `common.UserInfo`, etc.

#### `notifications/notification_service.proto` — Notification Service

This is the most interesting file — it imports from **both** `common/` **and** `orders/`, demonstrating **cross-service proto imports**.

```protobuf
syntax = "proto3";

package notifications;

import "common/base.proto";
import "common/user.proto";
import "common/pagination.proto";
import "orders/order_service.proto";     // ← cross-service import!

option java_package = "com.example.grpc.notifications";
option java_outer_classname = "NotificationServiceProto";
option java_multiple_files = true;

// ─────────────────────────────────────────────────
// Messages
// ─────────────────────────────────────────────────

message Notification {
  string notification_id = 1;
  common.UserInfo recipient = 2;          // ← from common/user.proto
  NotificationType type = 3;
  NotificationChannel channel = 4;
  string subject = 5;
  string body = 6;
  NotificationStatus status = 7;
  common.Timestamp created_at = 8;        // ← from common/base.proto
  common.Timestamp sent_at = 9;           // ← from common/base.proto
  common.Metadata metadata = 10;          // ← from common/base.proto

  // When the notification is order-related, embed the full order
  oneof context {
    orders.Order order_context = 11;       // ← from orders/order_service.proto!
    string custom_context_json = 12;
  }
}

enum NotificationType {
  NOTIFICATION_TYPE_UNSPECIFIED = 0;
  NOTIFICATION_TYPE_ORDER_CONFIRMATION = 1;
  NOTIFICATION_TYPE_ORDER_SHIPPED = 2;
  NOTIFICATION_TYPE_ORDER_DELIVERED = 3;
  NOTIFICATION_TYPE_ORDER_CANCELLED = 4;
  NOTIFICATION_TYPE_PAYMENT_RECEIVED = 5;
  NOTIFICATION_TYPE_REFUND_PROCESSED = 6;
  NOTIFICATION_TYPE_STOCK_ALERT = 7;
  NOTIFICATION_TYPE_PROMOTIONAL = 8;
}

enum NotificationChannel {
  NOTIFICATION_CHANNEL_UNSPECIFIED = 0;
  NOTIFICATION_CHANNEL_EMAIL = 1;
  NOTIFICATION_CHANNEL_SMS = 2;
  NOTIFICATION_CHANNEL_PUSH = 3;
  NOTIFICATION_CHANNEL_IN_APP = 4;
}

enum NotificationStatus {
  NOTIFICATION_STATUS_UNSPECIFIED = 0;
  NOTIFICATION_STATUS_QUEUED = 1;
  NOTIFICATION_STATUS_SENT = 2;
  NOTIFICATION_STATUS_DELIVERED = 3;
  NOTIFICATION_STATUS_FAILED = 4;
  NOTIFICATION_STATUS_READ = 5;
}

// ─────────────────────────────────────────────────
// Request / Response Messages
// ─────────────────────────────────────────────────

// Send a notification triggered by an order event
message SendOrderNotificationRequest {
  orders.Order order = 1;                       // ← cross-service type!
  NotificationType type = 2;
  repeated NotificationChannel channels = 3;    // send via multiple channels
}

// Send a custom (non-order) notification
message SendCustomNotificationRequest {
  common.UserInfo recipient = 1;
  NotificationType type = 2;
  NotificationChannel channel = 3;
  string subject = 4;
  string body = 5;
}

message GetNotificationRequest {
  string notification_id = 1;
}

message ListNotificationsRequest {
  string user_id = 1;
  NotificationType type_filter = 2;
  NotificationStatus status_filter = 3;
  common.PageRequest pagination = 4;           // ← from common/pagination.proto
}

message ListNotificationsResponse {
  repeated Notification notifications = 1;
  common.PageResponse pagination = 2;          // ← from common/pagination.proto
}

message SendNotificationResponse {
  string notification_id = 1;
  NotificationStatus status = 2;
  common.OperationStatus result = 3;           // ← enum from common/base.proto
}

// ─────────────────────────────────────────────────
// Service Definition
// ─────────────────────────────────────────────────

service NotificationService {
  // Send notification for an order event
  rpc SendOrderNotification(SendOrderNotificationRequest) returns (SendNotificationResponse);

  // Send a custom notification
  rpc SendCustomNotification(SendCustomNotificationRequest) returns (SendNotificationResponse);

  // Get a single notification by ID
  rpc GetNotification(GetNotificationRequest) returns (Notification);

  // List notifications with filtering and pagination
  rpc ListNotifications(ListNotificationsRequest) returns (ListNotificationsResponse);
}
```

**Highlight — Cross-service import:**
* `import "orders/order_service.proto"` brings in the full `Order` message
* `orders.Order` is used as a field type — the `orders.` prefix is required because it's from a different proto package
* The `oneof` block shows how to conditionally embed a cross-service type

## Using Generated Code in Java

### Importing Shared Types

With `java_multiple_files = true`, imports are clean and intuitive:

```java
// Common types — used everywhere
import com.example.grpc.common.Timestamp;
import com.example.grpc.common.Money;
import com.example.grpc.common.UserInfo;
import com.example.grpc.common.Address;
import com.example.grpc.common.PageRequest;

// Service-specific types
import com.example.grpc.orders.Order;
import com.example.grpc.orders.OrderItem;
import com.example.grpc.orders.OrderStatus;
import com.example.grpc.orders.OrderServiceGrpc;
```

### Building Nested Messages

When a message contains fields from imported types, you build them using nested builders:

```java
// Build shared types
Money unitPrice = Money.newBuilder()
        .setCurrencyCode("USD")
        .setUnits(29)
        .setNanos(990000000)  // $29.99
        .build();

Timestamp now = Timestamp.newBuilder()
        .setSeconds(System.currentTimeMillis() / 1000)
        .setNanos(0)
        .build();

UserInfo customer = UserInfo.newBuilder()
        .setUserId("user-42")
        .setUsername("alice")
        .setEmail("alice@example.com")
        .setRole(UserRole.USER_ROLE_CUSTOMER)
        .build();

Address shippingAddress = Address.newBuilder()
        .setStreetLine1("123 Main Street")
        .setCity("San Francisco")
        .setState("CA")
        .setPostalCode("94102")
        .setCountryCode("US")
        .build();

// Build a service-specific message using the shared types
OrderItem item = OrderItem.newBuilder()
        .setProductId("prod-001")
        .setProductName("Mechanical Keyboard")
        .setQuantity(2)
        .setUnitPrice(unitPrice)           // ← common.Money
        .setTotalPrice(Money.newBuilder()
                .setCurrencyCode("USD")
                .setUnits(59)
                .setNanos(980000000)       // $59.98
                .build())
        .build();

// Build the full Order
Order order = Order.newBuilder()
        .setOrderId("order-100")
        .setCustomer(customer)             // ← common.UserInfo
        .setShippingAddress(shippingAddress) // ← common.Address
        .setStatus(OrderStatus.ORDER_STATUS_PENDING)
        .addItems(item)
        .setSubtotal(unitPrice)
        .setTotal(unitPrice)
        .setCreatedAt(now)                 // ← common.Timestamp
        .setPaymentMethod(PaymentMethod.PAYMENT_METHOD_UPI)
        .build();
```

### Implementing a Server with Multiple Services

A single gRPC server can host **multiple services**:

```java
package com.example.grpc.server;

import io.grpc.Server;
import io.grpc.ServerBuilder;

public class ECommerceServer {

    private static final int PORT = 5005;

    public static void main(String[] args) throws Exception {
        Server server = ServerBuilder.forPort(PORT)
                .addService(new OrderServiceImpl())           // orders
                .addService(new InventoryServiceImpl())       // inventory
                .addService(new NotificationServiceImpl())    // notifications
                .build()
                .start();

        System.out.println("E-Commerce gRPC Server started on port " + PORT);
        System.out.println("  ✓ OrderService");
        System.out.println("  ✓ InventoryService");
        System.out.println("  ✓ NotificationService");

        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            System.out.println("Shutting down...");
            server.shutdown();
        }));

        server.awaitTermination();
    }
}
```

### Cross-Service Usage in Notification Servicer

The notification servicer shows how one service can reference types from another:

```java
package com.example.grpc.server;

import com.example.grpc.common.*;
import com.example.grpc.orders.Order;
import com.example.grpc.notifications.*;
import io.grpc.stub.StreamObserver;

public class NotificationServiceImpl
        extends NotificationServiceGrpc.NotificationServiceImplBase {

    @Override
    public void sendOrderNotification(
            SendOrderNotificationRequest request,
            StreamObserver<SendNotificationResponse> responseObserver) {

        // Access the Order from the orders package
        Order order = request.getOrder();

        // Access shared UserInfo from common package
        UserInfo customer = order.getCustomer();

        System.out.println("Sending " + request.getType() + " notification");
        System.out.println("  To: " + customer.getEmail());
        System.out.println("  Order: " + order.getOrderId());
        System.out.println("  Total: " + order.getTotal().getCurrencyCode()
                + " " + order.getTotal().getUnits()
                + "." + order.getTotal().getNanos());
        System.out.println("  Channels: " + request.getChannelsList());

        // Build response using common enum
        SendNotificationResponse response = SendNotificationResponse.newBuilder()
                .setNotificationId("notif-" + System.currentTimeMillis())
                .setStatus(NotificationStatus.NOTIFICATION_STATUS_SENT)
                .setResult(OperationStatus.OPERATION_STATUS_SUCCESS) // ← common enum
                .build();

        responseObserver.onNext(response);
        responseObserver.onCompleted();
    }
}
```

## Best Practices for Multi-Proto Projects

### 1. Use `java_multiple_files = true`

Always set this in every `.proto` file. Without it, all messages are nested inside an outer class, leading to awkward imports like `OrderServiceProto.Order` instead of just `Order`.

### 2. Consistent Package Naming

```
Proto package          →  Java package
common                 →  com.example.grpc.common
orders                 →  com.example.grpc.orders
inventory              →  com.example.grpc.inventory
notifications          →  com.example.grpc.notifications
```

### 3. Prefix Enum Values

Proto3 enums are **globally scoped** within a package. Always prefix values with the enum name to avoid collisions:

```protobuf
// ✅ Good — prefixed
enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
}

// ❌ Bad — collides with other enums in the same package
enum OrderStatus {
  UNSPECIFIED = 0;
  PENDING = 1;
}
```

### 4. Always Include an `UNSPECIFIED` Zero Value

Proto3 uses `0` as the default value. Always make it an explicit "unset" sentinel:

```protobuf
enum Status {
  STATUS_UNSPECIFIED = 0;  // sentinel — means "not set"
  STATUS_ACTIVE = 1;
}
```

### 5. Keep Import Direction Acyclic

```
common  ←──  orders  ←──  notifications
   ↑                          │
   └──────────────────────────┘   ✅ OK (all arrows point one way)

common  ←──  orders  ──►  notifications
                ↑               │
                └───────────────┘   ❌ CIRCULAR — will not compile
```

### 6. Separate Request/Response from Domain Messages

Keep your core domain messages (`Order`, `Product`) separate from RPC-specific wrappers (`CreateOrderRequest`, `ListOrdersResponse`). This makes the domain types cleanly reusable across services.

## Summary

| Concept | Description |
|---------|-------------|
| **`import`** | Pull types from another `.proto` file |
| **`package`** | Proto-level namespace for cross-file references |
| **`java_package`** | Controls the generated Java package |
| **`java_multiple_files`** | One `.java` file per message (cleaner imports) |
| **Cross-package refs** | Use `package.TypeName` (e.g., `common.Money`) |
| **Same-package refs** | Use `TypeName` directly (no prefix) |
| **Cross-service imports** | Import another service's proto to reuse its types |
| **Acyclic imports** | Import graph must be a DAG — no circular dependencies |
| **Gradle config** | No special config — plugin compiles all protos in `src/main/proto/` together |
