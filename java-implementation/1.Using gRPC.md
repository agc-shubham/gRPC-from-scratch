## How to Use gRPC with Java

### Introduction

In the previous modules, we learned the theory behind gRPC, Protocol Buffers, and how to implement a gRPC service in Python. In this module, we will implement the **same concepts using Java** — one of the most widely used languages for gRPC in production systems.

Java's strong type system pairs naturally with protobuf's strict typing, making it an excellent fit for gRPC services.

---

## Using gRPC With Java

Implementing gRPC with Java involves three core dependencies:

* `protobuf-java` — the Protocol Buffers runtime for Java
* `grpc-netty-shaded` (or `grpc-netty`) — the gRPC transport layer
* `grpc-protobuf` — bridges protobuf and gRPC
* `grpc-stub` — provides client stub classes

Unlike Python where we manually run `grpc_tools.protoc`, Java projects use a **build plugin** (Gradle or Maven) that automatically generates Java code from `.proto` files during the build.

```
┌──────────────┐   protobuf-gradle-plugin   ┌──────────────────┐    grpc-java     ┌──────────────────┐
│  .proto file │  ────────────────────────►  │  Generated Java  │ ──────────────►  │  Client & Server │
│ (Definitions)│    (or Maven plugin)        │  Classes & Stubs │                  │  Application Code│
└──────────────┘                             └──────────────────┘                  └──────────────────┘
```

## Creating a gRPC Client and Server in Java

### Step 1: Project Setup

We'll use **Gradle** for this guide. Create a standard Gradle project structure:

```
grpc-java-demo/
├── build.gradle
├── settings.gradle
└── src/
    ├── main/
    │   ├── java/
    │   │   └── com/example/grpc/
    │   │       ├── ItemServer.java
    │   │       └── ItemClient.java
    │   └── proto/
    │       └── item.proto
    └── test/
        └── java/
```

### Step 2: Configure `build.gradle`

Add the required dependencies and the protobuf compiler plugin:

```groovy
plugins {
    id 'java'
    id 'application'
    id 'com.google.protobuf' version '0.9.4'
}

group = 'com.example'
version = '1.0-SNAPSHOT'
sourceCompatibility = '11'

repositories {
    mavenCentral()
}

def grpcVersion = '1.68.0'
def protobufVersion = '4.28.0'

dependencies {
    // gRPC dependencies
    implementation "io.grpc:grpc-netty-shaded:${grpcVersion}"
    implementation "io.grpc:grpc-protobuf:${grpcVersion}"
    implementation "io.grpc:grpc-stub:${grpcVersion}"

    // Required for Java 9+
    compileOnly 'javax.annotation:javax.annotation-api:1.3.2'
}

protobuf {
    protoc {
        artifact = "com.google.protobuf:protoc:${protobufVersion}"
    }
    plugins {
        grpc {
            artifact = "io.grpc:protoc-gen-grpc-java:${grpcVersion}"
        }
    }
    generateProtoTasks {
        all()*.plugins {
            grpc {}
        }
    }
}
```

Also create a minimal `settings.gradle`:

```groovy
rootProject.name = 'grpc-java-demo'
```

### Step 3: Define the `.proto` File

Create `src/main/proto/item.proto`:

```protobuf
syntax = "proto3";

option java_package = "com.example.grpc";
option java_outer_classname = "ItemProto";

message ItemMessage {
  string name = 1;
  string brand_name = 2;
  int32 id = 3;
  float weight = 4;
}

service ItemService {
    rpc Create(ItemMessage) returns (ItemMessage);
}
```

**Java-specific options:**
* `java_package` — sets the Java package for generated classes
* `java_outer_classname` — sets the wrapper class name for messages

---

### Step 4: Generate gRPC Files

Run the Gradle build to auto-generate Java files:

```bash
gradle build
```

This generates two key files in `build/generated/source/proto/main/`:
* `ItemProto.java` — message classes (equivalent to Python's `item_pb2.py`)
* `ItemServiceGrpc.java` — service stubs (equivalent to Python's `item_pb2_grpc.py`)

> ⚠️ **Do not edit** the generated files — they are recreated on every build.

---

### Step 5: Implement the gRPC Server

Create `src/main/java/com/example/grpc/ItemServer.java`:

```java
package com.example.grpc;

import io.grpc.Server;
import io.grpc.ServerBuilder;
import io.grpc.stub.StreamObserver;

import java.io.IOException;
import java.util.logging.Logger;

public class ItemServer {

    private static final Logger logger = Logger.getLogger(ItemServer.class.getName());
    private static final int PORT = 5005;

    public static void main(String[] args) throws IOException, InterruptedException {
        Server server = ServerBuilder.forPort(PORT)
                .addService(new ItemServiceImpl())
                .build()
                .start();

        logger.info("gRPC Server started on port " + PORT);

        // Keep server running until shutdown
        server.awaitTermination();
    }

    /**
     * Implementation of the ItemService defined in item.proto.
     * This is equivalent to Python's ItemServicer class.
     */
    static class ItemServiceImpl extends ItemServiceGrpc.ItemServiceImplBase {

        @Override
        public void create(ItemProto.ItemMessage request,
                           StreamObserver<ItemProto.ItemMessage> responseObserver) {

            // Log the received message
            System.out.println("Received item:");
            System.out.println("  Name: " + request.getName());
            System.out.println("  Brand: " + request.getBrandName());
            System.out.println("  ID: " + request.getId());
            System.out.println("  Weight: " + request.getWeight());

            // Return the same message (echo behavior)
            responseObserver.onNext(request);
            responseObserver.onCompleted();
        }
    }
}
```

**Key differences from Python:**
* Java uses `StreamObserver<T>` for responses instead of returning directly
* `responseObserver.onNext()` sends the response
* `responseObserver.onCompleted()` signals end of RPC
* The servicer extends `ItemServiceGrpc.ItemServiceImplBase` (auto-generated base class)

### Step 6: Implement the gRPC Client

Create `src/main/java/com/example/grpc/ItemClient.java`:

```java
package com.example.grpc;

import io.grpc.ManagedChannel;
import io.grpc.ManagedChannelBuilder;

import java.util.concurrent.TimeUnit;
import java.util.logging.Logger;

public class ItemClient {

    private static final Logger logger = Logger.getLogger(ItemClient.class.getName());
    private static final String HOST = "localhost";
    private static final int PORT = 5005;

    public static void main(String[] args) throws InterruptedException {
        // Create a channel to the server
        ManagedChannel channel = ManagedChannelBuilder
                .forAddress(HOST, PORT)
                .usePlaintext()  // No TLS for local development
                .build();

        // Create a blocking stub (synchronous client)
        ItemServiceGrpc.ItemServiceBlockingStub stub =
                ItemServiceGrpc.newBlockingStub(channel);

        // Build the ItemMessage
        ItemProto.ItemMessage item = ItemProto.ItemMessage.newBuilder()
                .setName("Hair Dryer")
                .setBrandName("Dry King")
                .setId(34)
                .setWeight(1.2f)
                .build();

        // Call the Create RPC
        ItemProto.ItemMessage response = stub.create(item);

        System.out.println("Response received:");
        System.out.println("  Name: " + response.getName());
        System.out.println("  Brand: " + response.getBrandName());
        System.out.println("  ID: " + response.getId());
        System.out.println("  Weight: " + response.getWeight());

        // Shutdown the channel gracefully
        channel.shutdown().awaitTermination(5, TimeUnit.SECONDS);
    }
}
```

**Key differences from Python:**
* Java uses `ManagedChannel` instead of `grpc.insecure_channel()`
* Messages are built using the **Builder pattern**: `ItemMessage.newBuilder()...build()`
* **Blocking stub** = synchronous calls (like Python's default), **Future stub** and **Async stub** are also available
* Fields use camelCase getters/setters (`getBrandName()`) even though the proto uses snake_case (`brand_name`)

### Python vs Java — Side by Side

| Aspect | Python | Java |
|--------|--------|------|
| **Dependencies** | `grpcio`, `grpcio-tools` | `grpc-netty-shaded`, `grpc-protobuf`, `grpc-stub` |
| **Code generation** | Manual `grpc_tools.protoc` command | Automatic via Gradle/Maven plugin |
| **Generated files** | `_pb2.py`, `_pb2_grpc.py` | `*Proto.java`, `*ServiceGrpc.java` |
| **Server base class** | `ServiceServicer` | `ServiceGrpc.ServiceImplBase` |
| **Response mechanism** | `return message` | `responseObserver.onNext()` + `.onCompleted()` |
| **Client channel** | `grpc.insecure_channel()` | `ManagedChannelBuilder.forAddress()` |
| **Message creation** | Constructor kwargs | Builder pattern |
| **Type enforcement** | Runtime errors | Compile-time errors |

### Running the Demo

```bash
# Terminal 1 — Start the gRPC server
gradle run --args="server"
# Or run the class directly:
gradle runServer
```

Alternatively, build a fat JAR and run directly:

```bash
gradle build

# Terminal 1
java -cp build/libs/grpc-java-demo-1.0-SNAPSHOT.jar com.example.grpc.ItemServer

# Terminal 2
java -cp build/libs/grpc-java-demo-1.0-SNAPSHOT.jar com.example.grpc.ItemClient
```

You can also add custom Gradle tasks for convenience in `build.gradle`:

```groovy
task runServer(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.ItemServer'
}

task runClient(type: JavaExec) {
    classpath = sourceSets.main.runtimeClasspath
    mainClass = 'com.example.grpc.ItemClient'
}
```

Then simply run:

```bash
# Terminal 1
gradle runServer

# Terminal 2
gradle runClient
```

### Showcasing Type Checking

Java's compile-time type checking goes even further than Python's runtime checking:

```java
// This will NOT compile — caught at build time!
ItemProto.ItemMessage item = ItemProto.ItemMessage.newBuilder()
        .setBrandName(12345)  // ❌ Compile error: expected String, got int
        .build();
```

In Python, this error would only appear at **runtime**. In Java, the compiler catches it **before the code ever runs**.

### New Terms

| Term | Definition |
|------|-----------|
| `grpc-netty-shaded` | Java gRPC transport implementation based on Netty (HTTP/2 server) |
| `grpc-protobuf` | Bridges Protocol Buffers with Java gRPC |
| `grpc-stub` | Provides generated stub classes for gRPC clients |
| `StreamObserver` | Java interface for handling gRPC streaming responses |
| `ManagedChannel` | A channel to the gRPC server, managing the underlying HTTP/2 connection |
| `BlockingStub` | Synchronous client stub — blocks until the server responds |
| Builder Pattern | Java pattern used to construct immutable protobuf messages |
